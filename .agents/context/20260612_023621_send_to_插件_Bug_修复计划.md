# send_to 插件 Bug 修复计划

## 发现的 Bug 和问题

### Bug 1: `wander.py` 导入路径错误（严重）
- **位置**: `wander.py` 第 27-31 行
- **问题**: 导入了 `src.core.components.base.event_handler.BaseEventHandler` 和 `src.kernel.llm.ROLE, LLMPayload, Text`，而插件其他所有文件统一使用 `src.app.plugin_system.base.BaseEventHandler` 和 `src.app.plugin_system.types.ROLE, LLMPayload, Text`。这会导致：
  1. `BaseEventHandler` 可能不是同一个类，导致 `isinstance` 检查失败或方法签名不一致
  2. `ROLE` / `LLMPayload` / `Text` 可能不是同一套类型，导致 LLM 请求构建失败
- **修复**: 将导入路径改为与插件其他文件一致：
  - `from src.app.plugin_system.base import BaseEventHandler`
  - `from src.app.plugin_system.types import ROLE, LLMPayload, Text`

### Bug 2: `auto_inject.py` KFC/非KFC分支几乎完全重复（代码质量）
- **位置**: `auto_inject.py` 第 534-558 行
- **问题**: `_build_injection_text` 函数的 KFC 和非 KFC 分支文本几乎完全相同，唯一区别是 KFC 分支末尾多了"但不要直接提及或引用这些内容，除非对方主动提起。"这句话。这是明显的代码重复。
- **修复**: 合并两个分支，KFC 模式追加额外提示即可。

### Bug 3: `stream_index.py` `upsert_summary` 使用 `hasattr` 检查（代码质量）
- **位置**: `stream_index.py` 第 427 行
- **问题**: `if hasattr(chat_stream, "message") and chat_stream.message:` — 按照代码规范，应避免 `hasattr` 防御式编程。`chat_stream` 对象的 `message` 属性是确定的，应直接用 `getattr` 或直接访问。
- **修复**: 改为 `if getattr(chat_stream, "message", None):`

### Bug 4: `context_lookup.py` `around_user` 参数类型判断逻辑冗余
- **位置**: `context_lookup.py` 第 484 行
- **问题**: `around_user if around_user is not None else config.lookup.around_user_default` — 但 `SendToUserContextTool.execute` 的参数 `around_user: bool = True`，默认值是 `True`，永远不会是 `None`。所以 `is not None` 检查永远为 `True`，配置默认值永远不会被使用。
- **修复**: 将 `SendToUserContextTool` 的 `around_user` 默认值改为 `None`，或直接使用 `around_user or config.lookup.around_user_default`。考虑到 Tool 参数默认值语义，改为 `around_user: bool | None = None` 更合理。

### Bug 5: `daily_memory.py` `_generate_full_day_summary` 增量模式返回旧记录时未更新水位线
- **位置**: `daily_memory.py` 第 314-324 行
- **问题**: 当增量模式下 `raw_messages` 为空时（即水位线之后没有新消息），函数直接返回 `existing_record`。这本身是正确的。但问题在于：如果 `existing_record.last_summarized_ts` 是 0（旧数据迁移场景），且当日无消息，函数会返回一个 `last_summarized_ts=0` 的记录，后续增量更新会从 0 开始拉取所有消息，导致重复总结。
- **严重程度**: 低（边界情况，仅影响旧数据迁移）
- **修复**: 在返回 `existing_record` 之前，如果 `last_summarized_ts` 为 0，将其更新为当日 0 点时间戳，避免下次全量拉取。

### Bug 6: `wander.py` `_llm_decide` 中 `model_set` 被当作 list 处理
- **位置**: `wander.py` 第 298-306 行
- **问题**: `model_set = llm_api.get_model_set_by_task("sub_actor")` 返回的类型被当作 list 来做列表推导式修改。但 `get_model_set_by_name` 返回的可能是不同类型。如果 `get_model_set_by_task` 返回的不是 list，这里会报错。同时，直接修改 model_set 中的 temperature/max_tokens 可能不是正确的 API 用法。
- **严重程度**: 中（取决于 `llm_api` 的实际返回类型）
- **修复**: 需要确认 `llm_api.get_model_set_by_task` 的返回类型。如果它返回的是不可变对象或 dict，需要使用正确的 API 来覆盖参数。参考 `stream_index.py` 和 `daily_memory.py` 中的用法，它们只用了 `get_model_set_by_task` 和 `create_llm_request`，没有手动修改 model_set。

### Bug 7: `commands.py` `SendToMemoryCommand._current_chat_type` 从 `extra` 取 `chat_type`
- **位置**: `commands.py` 第 35-39 行
- **问题**: `self._message.extra.get("chat_type")` — 但 `Message` 对象本身就有 `chat_type` 属性，不应该从 `extra` 字典中取。如果 `extra` 中没有 `chat_type` 键，会返回空字符串，导致群聊判断失败。
- **修复**: 改为 `self._message.chat_type if self._message else ""`

### Bug 8: `action.py` 第 521 行多余空行
- **位置**: `action.py` 第 521 行
- **问题**: `SendToSummaryUpdateAction` 类定义前有两个空行，不符合 PEP 8（顶层定义间应有两个空行，但类内方法间只需一个）。
- **严重程度**: 极低（格式问题）
- **修复**: 删除多余空行。

## 修复优先级

2. **Bug 7** (commands.py chat_type 取法) — 中等，可能导致命令判断错误
3. **Bug 6** (wander.py model_set 处理) — 中等，可能导致串门功能异常
4. **Bug 4** (around_user 默认值) — 低，配置默认值永远不生效
5. **Bug 2** (auto_inject 代码重复) — 代码质量
6. **Bug 3** (hasattr 防御式编程) — 代码质量
7. **Bug 5** (水位线边界) — 低，边界情况
8. **Bug 8** (格式) — 极低
