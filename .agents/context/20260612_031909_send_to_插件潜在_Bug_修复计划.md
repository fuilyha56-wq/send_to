## Bug 3: action.py 去重逻辑丢失信息（中等）

**文件**: `action.py`
**问题**: `list({gid: gname for gid, gname in exact}.items())` 用 dict 去重时，同一 gid 不同 gname 会丢失。应保留第一个匹配。
**修复**:
- 将 exact/partial 去重改为保留首次出现的逻辑：
```python
seen_gids: set[str] = set()
unique_exact: list[tuple[str, str]] = []
for gid, gname in exact:
    if gid not in seen_gids:
        seen_gids.add(gid)
        unique_exact.append((gid, gname))
```
- partial 同理

**验收**: 同一 gid 多个 gname 时保留第一个，不丢失。

## Bug 4: context_lookup.py scope 降级逻辑不合理（中等）

**文件**: `context_lookup.py`
**问题**: 用户请求 `group` 但不在 allowed 中时，代码按 `_ALL > _PRIVATE > _GROUP` 优先级降级，可能返回与用户意图不符的 scope。
**修复**:
- 当请求的 scope 不在 allowed 中时，应按与请求 scope 的"接近度"降级：
  - 请求 `group` → 尝试 `all`（包含 group），再 `group`，再 `private`
  - 请求 `private` → 尝试 `all`（包含 private），再 `private`，再 `group`
  - 请求 `all` → 尝试 `all`，再 `private`，再 `group`
- 简化实现：如果请求的 scope 不在 allowed 中，按 `{请求scope, all, 其他}` 的优先级尝试

```python
def normalize_chat_scope(chat_scope: str, config: SendToConfig) -> str:
    normalized = str(chat_scope or _ALL).strip().lower()
    if normalized not in {_PRIVATE, _GROUP, _ALL}:
        normalized = _ALL
    allowed = normalize_list(config.privacy.allowed_chat_scopes)
    if not allowed or normalized in allowed:
        return normalized
    # 按与请求 scope 的相关性降级
    fallback_order = {
        _ALL: [_ALL, _PRIVATE, _GROUP],
        _PRIVATE: [_ALL, _PRIVATE, _GROUP],
        _GROUP: [_ALL, _GROUP, _PRIVATE],
    }
    for candidate in fallback_order.get(normalized, [_ALL, _PRIVATE, _GROUP]):
        if candidate in allowed:
            return candidate
    return ""
```

**验收**: 请求 group 时优先降级到 all（包含 group），而非 private。

## Bug 5: _stream_locks / _state_locks 无界增长（低）

**文件**: `stream_index.py`, `daily_memory.py`
**问题**: 模块级 Lock 字典永不清理，长期运行可能积累大量无用 Lock。
**修复**:
- 在获取锁时增加简单清理：当字典大小超过阈值（如 256）时，删除未被持有的锁
- 实现方式：在 `_get_stream_lock` / `_state_lock` 中，当 `len(dict) > 256` 时，过滤掉 `not lock.locked()` 的条目

```python
def _get_stream_lock(stream_id: str) -> asyncio.Lock:
    lock = _stream_locks.get(stream_id)
    if lock is None:
        # 简单清理：防止无界增长
        if len(_stream_locks) > 256:
            _stream_locks = {
                k: v for k, v in _stream_locks.items() if v.locked()
            }
        lock = asyncio.Lock()
        _stream_locks[stream_id] = lock
    return lock
```

**验收**: Lock 字典不会无限增长。

## Bug 7: auto_inject.py 群聊 person_id 回退到 stream 级占位值（中等）

**文件**: `auto_inject.py`
**问题**: 群聊中如果无法从 values/recent_msgs 解析出触发用户 person_id，会回退到 `ChatStreams.person_id`，而群聊的该字段可能只是占位值，导致跨流查询结果不正确。
**修复**:
- 在 `_resolve_effective_person_id` 中，群聊场景下如果 trigger_person_id 和 recent_msgs 都无法解析出 person_id，返回空字符串而非 stream_person_id，让调用方跳过注入而非用错误 person_id 查询

```python
def _resolve_effective_person_id(...) -> str:
    stream_person_id = _normalize_text(getattr(current_stream, "person_id", ""))
    trigger_person_id = _extract_trigger_person_id(values)
    if chat_type == "group":
        if trigger_person_id:
            return trigger_person_id
        resolved_person_id = _resolve_trigger_person_id_from_messages(...)
        return resolved_person_id  # 不再回退到 stream_person_id
    return trigger_person_id or stream_person_id
```

**验收**: 群聊中无法解析 person_id 时返回空串，调用方跳过注入。

## Bug 8: SendToExecuteAction 重复解析 group_id（低）

**文件**: `action.py`
**问题**: `_resolve_stream_id` 已解析 group_id，但 `SendToExecuteAction.execute` 又重新调用 `_resolve_group_id`，可能得到不同结果且浪费查询。
**修复**:
- 从已生成的 `target_stream_id` 中提取 group_id，或直接使用已知的 group_id 参数
- 简化：如果 `group_id` 已提供直接使用，否则从 target_stream_id 对应的 stream info 中获取

```python
# 替换 446-453 行
extra: dict[str, Any] = {}
if normalized_type == "group":
    resolved_group_id = str(group_id or "").strip()
    if not resolved_group_id and group_hint:
        # group_hint 已在 _resolve_stream_id 中被使用，此处从 stream info 获取
        if target_info:
            resolved_group_id = str(target_info.get("group_id", "") or "")
    if resolved_group_id:
        extra["target_group_id"] = resolved_group_id
```

**验收**: 不重复调用 `_resolve_group_id`，从已有信息中获取 group_id。

## Bug 9: daily_memory.py 跨天归档后 last_summary_at 重置为 0（中等）

**文件**: `daily_memory.py`
**问题**: 跨天归档后将 `last_summary_at` 重置为 `0.0`，导致新一天只能靠轮次触发，idle 触发永远不满足。如果群消息很少，短期记忆可能永远不会更新。
**修复**:
- 跨天归档后将 `last_summary_at` 设为当前时间而非 0，这样 idle 触发逻辑在新一天也能正常工作

```python
# register_bot_message 中 576-579 行
state.current_date = today
state.round_count = 0
state.last_summary_at = time.time()  # 改为当前时间
state.last_event_direction = ""
```

- 同样修复 `archive_yesterday_for_all` 中 694-697 行

**验收**: 跨天后 idle 触发条件能正常满足。
