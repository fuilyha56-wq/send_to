# Plan: 支持向无聊天流记录的用户发送消息

## Goal

修复 send_to 插件无法向没有聊天流记录的用户发送私聊消息的问题，同时增强 `send_to_find_stream` 工具使其能按 QQ 号/群号直接查找目标流。

## Current Understanding

### Current behavior
- `SendToAction` 调用 `_resolve_stream_id()` 生成 SHA256 stream_id 后直接调用 `send_text()`
- `send_text()` → `_send_message()` 需要从 `stream_manager.get_stream_info()` 获取 `person_id` 来反查 `target_user_id`
- 若 `ChatStreams` 表中无该流记录 → `stream_info` 为 None → `target_user_id` 无法注入 `Message.extra`
- OneBot 适配器的 `send_normal_message()` 检查 `user_info.get("user_id")` → 为 bot 自身 ID → 发送失败
- `SendToFindStreamTool` 仅搜索 `list_summary_records()`，无历史记录的用户不可见

### Expected behavior
- 向已知 QQ 号的用户发送私聊时，自动创建流记录，消息成功送达
- 按 QQ 号/群号查找目标流时，即使无摘要记录也能找到

### Relevant context
- `stream_api.get_or_create_stream()` 已存在且被 `relay_intent()` 使用，证明方案可行
- SnowLuma/OneBot 的 `send_private_msg` 可直接向任意 QQ 号发送，无需好友关系
- `_resolve_stream_id` 内部已计算出 `normalized_user_id` / `normalized_group_id`，但未返回给调用方

### Confirmed files / interfaces / entry points
- `action.py` — `_resolve_stream_id()` (L165-215), `SendToAction.execute()` (L233-305), `SendToExecuteAction.execute()` (L337-450+)
- `tools.py` — `SendToFindStreamTool.execute()` (L246-284)
- `src/app/plugin_system/api/stream_api.py` — `get_or_create_stream()`
- `src/core/models/stream.py` — `ChatStream.generate_stream_id()`

## Scope

### In Scope
1. 修改 `_resolve_stream_id` 返回值，增加 resolved_id
2. `SendToAction` 和 `SendToExecuteAction` 在解析后调用 `get_or_create_stream()` 确保流记录存在
3. `SendToFindStreamTool` 增加纯数字标识符回退路径

### Out of Scope
- 不修改 `manifest.json` 版本号
- 不修改配置文件
- 不修改 relay / wander / auto_inject 等其他模块
- 不添加新的 Tool 或 Action 类型
- 不修改 OneBot 适配器或框架核心代码

## Implementation Steps

### Step 1: 修改 `_resolve_stream_id` 返回值

**Expected behavior:**
- 返回 `(stream_id, error, resolved_id)` 三元组
- `resolved_id` 为解析后的 group_id（群聊）或 user_id（私聊）
- 错误路径返回 `(None, error_msg, "")`

**Files / entry points:**
- `action.py` L173: 修改返回类型注解
- `action.py` L165-215: 所有 return 语句

**Implementation notes:**
- 类型注解改为 `tuple[str | None, str, str]`
- 群聊路径：`resolved_id = normalized_group_id`
- 私聊路径：`resolved_id = normalized_user_id`
- 错误路径：第三个元素返回空字符串 `""`

**Acceptance criteria:**
- 所有 return 语句包含第三个元素
- 类型注解正确

### Step 2: 更新 `SendToAction.execute()` 调用方

**Expected behavior:**
- 解包三元组
- 解析成功后调用 `stream_api.get_or_create_stream()` 确保流记录存在
- 发送行为不变

**Files / entry points:**
- `action.py` L276-283: 解包和流创建

**Implementation notes:**
```python
target_stream_id, err, resolved_id = await _resolve_stream_id(...)
if err:
    yield False, err
    return

# 确保流记录存在（支持向无聊天记录的用户发送）
try:
    from src.app.plugin_system.api import stream_api
    if normalized_type == "group":
        await stream_api.get_or_create_stream(
            stream_id=target_stream_id,
            platform=effective_platform,
            group_id=resolved_id,
            chat_type="group",
        )
    else:
        await stream_api.get_or_create_stream(
            stream_id=target_stream_id,
            platform=effective_platform,
            user_id=resolved_id,
            chat_type="private",
        )
except Exception:
    pass  # 非致命：send_text 可能仍能工作
```

**Acceptance criteria:**
- 私聊发送到无流记录用户时，流自动创建
- 已有流的用户行为不变
- 异常不阻断发送流程

### Step 3: 更新 `SendToExecuteAction.execute()` 调用方

**Expected behavior:**
- 同 Step 2，解包三元组并创建流

**Files / entry points:**
- `action.py` L442-453: 解包和流创建

**Implementation notes:**
- 与 Step 2 相同的模式
- 在 `if err:` 检查之后、`yield None` 之前插入

**Acceptance criteria:**
- 跨流执行 Action 到无流记录用户时，流自动创建

### Step 4: 增强 `SendToFindStreamTool` 数字标识符回退

**Expected behavior:**
- 当 `identifier` 为纯数字但在 summary_records 中找不到时：
  - 根据 `chat_type_hint` 或尝试两种类型生成 stream_id
  - 调用 `stream_api.get_or_create_stream()` 确保流存在
  - 返回目标流信息（标注无摘要）

**Files / entry points:**
- `tools.py` L269-278: 纯数字匹配失败后的回退逻辑

**Implementation notes:**
- 在 `elif ident.isdigit():` 分支中，`target` 为 None 时增加回退
- 使用 `ChatStream.generate_stream_id()` 生成 stream_id
- 调用 `stream_api.get_or_create_stream()` 确保记录存在
- 返回格式与现有成功路径一致，summary 标注 "(暂无摘要)"

**Acceptance criteria:**
- `send_to_find_stream(identifier="123456")` 对无历史用户返回有效结果
- 已有摘要的用户行为不变
- 歧义情况（同号可能是群也可能是用户）给出提示

## Edge Cases

1. **user_id 无效**：OneBot `send_private_msg` 返回错误，正常传播给 LLM
2. **流已存在**：`get_or_create_stream` 幂等，安全
3. **并发创建**：`get_or_create_stream` 内部使用 per-stream asyncio.Lock
4. **user_hint 解析**：`_resolve_stream_id` 内部已调用 `_resolve_user_id`，resolved_id 为解析后的 user_id
5. **群聊场景**：群聊流通常由框架自动创建，此修改作为安全兜底
